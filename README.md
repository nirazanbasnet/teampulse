## Setup order

### 1. Create Supabase project

```bash
# Install Supabase CLI
npm install -g supabase

# Login
supabase login

# Link to your project
supabase link --project-ref YOUR_PROJECT_REF

# Run migrations
supabase db push supabase/migrations/001_initial_schema.sql
supabase db push supabase/policies/002_rls_policies.sql

# Generate TypeScript types
supabase gen types typescript --linked > src/lib/supabase/database.types.ts
```

### 2. Install dependencies

```bash
npm install @supabase/supabase-js @supabase/ssr
npm install @dnd-kit/core @dnd-kit/sortable @dnd-kit/utilities
npm install recharts
npm install @anthropic-ai/sdk
npm install next@latest react@latest react-dom@latest
# shadcn/ui
npx shadcn-ui@latest init
```

### 3. Environment variables

```env
# .env.local

# Supabase (public — safe for client)
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key

# Supabase service role — SERVER ONLY, never expose to client
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key

# Anthropic — SERVER ONLY
ANTHROPIC_API_KEY=sk-ant-...
```

### 4. Enable Supabase Realtime

In the Supabase dashboard:
- Go to Database → Replication
- Enable replication for the `notes` table
- Set replica identity to `FULL` so UPDATE payloads include old values:

```sql
alter table public.notes replica identity full;
```

### 5. Vercel deployment

```bash
# Install Vercel CLI
npm install -g vercel

# Deploy
vercel --prod

# Set environment variables in Vercel dashboard
# NEVER commit .env.local to git
```

## Key security decisions

### Author anonymity is enforced in layers

1. **Database constraint**: `no_self_feedback` CHECK prevents self-notes at DB level
2. **RLS policy**: `notes_safe` VIEW omits `author_id` entirely from the SELECT projection
3. **Timing obfuscation**: `created_at_display` is hour-rounded in a generated column
4. **Server action validation**: `createNote` re-checks `author_id != recipient_id` before insert
5. **Client never receives author_id**: `BoardView` and `NoteCard` have no `author_id` prop

### Service role is strictly server-side

- `createServiceClient()` throws if called without `SUPABASE_SERVICE_ROLE_KEY`
- Only used in two places: `summaries.ts` (AI generation) and `notes.ts` moderation
- The `notes_admin` view (which includes `author_id`) is only queryable with service role

### Minimum team size for analytics

The `settings.min_team_size_for_anon` workspace setting (default 3) should be checked
before rendering per-author tag breakdowns, preventing a two-member team from inferring
authorship via process of elimination. Implement this check in `analytics/[teamId]/page.tsx`.

## Supabase Realtime channel design

One channel per team board:

```
board:{teamId}
  └── postgres_changes → notes table, filter: team_id=eq.{teamId}
```

The `notes_safe` view is NOT used for Realtime — Realtime fires on the base table.
The `author_id` field in Realtime payloads is stripped in `use-board-realtime.ts`
before updating React state. This is a critical security step — do not remove it.

## AI summary generation flow

```
Admin clicks "Close cycle"
  → closeCycleAndGenerateSummaries(cycleId) server action
  → Load all notes via notes_admin view (service role)
  → For each recipient with >= 2 notes:
      → Call Claude claude-sonnet-4-20250514
      → Prompt strips author attribution language
      → Store result in cycle_summaries (recipient_id, cycle_id)
  → Update cycle status to 'closed'
  → Member logs in → reads own summary from cycle_summaries
     (RLS: profile_id = auth.uid() only)
```

Summaries are never shown to admins, managers, or other team members.
