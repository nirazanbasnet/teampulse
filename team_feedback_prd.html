Product Requirements Document

TeamPulse — Collaborative anonymous team feedback platform

A kanban-style board where team members leave anonymous sticky-note feedback on each other, track completion, and surface growth metrics over time.

v1.0 · June 2026
Internal Teams
Status: Draft
Web App

Problem statement

Most teams lack a lightweight, safe channel for continuous peer feedback. Annual reviews are infrequent; verbal feedback is inconsistent; named feedback creates social friction.

Feedback is rare and high-stakes

Without a low-friction channel, feedback only surfaces at formal reviews — too late to course-correct and too formal to be candid.

Social fear suppresses honest input

Members holding back concerns because they fear damaging relationships. Anonymity removes that barrier and raises the quality of input.

No visibility into team growth over time

Feedback happens but is never aggregated. Teams can't spot recurring themes, track improvement, or measure engagement with feedback received.

Target users

Two primary roles with distinct needs.

AD

Admin / Team lead

Sets up teams, manages membership, views aggregate metrics, and configures feedback cycles.

  • Create and name teams
  • Add / remove members
  • View all feedback in aggregate
  • Export reports
TM

Team member

Gives anonymous feedback to colleagues, receives feedback on their own column, marks notes as actioned.

  • Write feedback on others
  • View feedback received
  • Mark notes as done
  • See own growth metrics

Core design principles

Anonymity first

The identity of a feedback author is never exposed to the recipient — not in the UI, not in metadata, not inferred by timing. Anonymity is enforced server-side, not just hidden in the UI.

Tactile and spatial

Sticky notes as the primary metaphor — draggable, colour-coded, with a physical "done" state. The board should feel alive, not like a form.

Data as a growth tool

Aggregate patterns surface themes across the team. Members own their own data and use it for self-improvement, not performance management.

You cannot review yourself

A user's own column is read-only to them. They can see feedback received but cannot add notes to their own column under any circumstance.

Feature catalogue

Organised by module. Priority uses the MoSCoW framework — must have, should have, could have.

Auth & team management
User authentication

Email/password login and magic-link passwordless auth. Session persistence with JWT. Role flag at account level: admin or member.

Must
Team creation by admin

Admin creates a named team (e.g., "Frontend team"). Teams are isolated — feedback is scoped within a team and not visible across teams.

Must
Member assignment

Admin assigns registered users to a team. A user may belong to multiple teams. Removing a member hides their column from active view but preserves historical data.

Must
Invite by email

Admin sends an invite link to a new user. On accepting, they are auto-assigned to the team. Invite expires after 7 days.

Should
Feedback board
Kanban columns per member

Each team member gets a named column. Columns show member name and avatar (initials). Columns are horizontally scrollable on the board.

Must
Anonymous sticky note creation

Any member can click "+ Add feedback" on another member's column. Note is stored with author ID server-side but displayed without any authorship indication to the recipient. Colour-coded by type: yellow = general, green = strength, blue = growth area.

Must
Draggable notes

Notes within a column can be repositioned vertically via drag-and-drop. Supports reordering by priority or category. Uses dnd-kit or native HTML drag API.

Must
Mark note as done

The recipient can mark any note on their own column as "done". Done notes collapse visually with a strikethrough and a green completion chip. They can be expanded to review later.

Must
Self-column write lock

The add-note action is disabled on the logged-in user's own column. The column header shows a "Your column" chip and the input area is visually locked.

Must
Edit / delete own notes

The author can edit or delete their own submitted notes (within a grace period, e.g., 30 minutes). After the grace period, notes are locked.

Should
Note tagging

Notes can be tagged with predefined labels: Communication, Technical, Collaboration, Leadership, Delivery. Tags are visible to everyone and drive the metrics module.

Should
Reactions on notes

Members can add emoji reactions to a note (👍 💡 ❤️). Reaction counts are visible, identities are not. Acts as lightweight signal amplification.

Could
Analytics & metrics
Personal feedback dashboard

Each member sees: notes received over time, completion rate, tag breakdown (radar/bar), and a streak counter for responding to feedback.

Must
Team admin analytics view

Admin sees team-level: total notes given/received, average completion rate, most-used tags, participation rate (% of members who gave feedback this cycle).

Must
Feedback cycles

Admin can open and close named feedback cycles (e.g., "Sprint 42", "Q2 Retro"). Notes belong to a cycle. Metrics are filterable by cycle.

Should
Export to PDF / CSV

Admin can export a team's feedback summary for a cycle as a PDF report or raw CSV. Exports never include author identity — only aggregated or anonymous note content.

Should
AI theme summariser

At cycle close, use Claude API to generate a thematic summary per member: "Your peers consistently highlight your communication and flag delivery timelines as a growth area." Displayed only to the member themselves.

Could
Notifications
In-app notifications

Member is notified when new feedback appears on their column (never revealing author). Admin is notified when participation drops below a threshold.

Must
Email digest

Weekly or per-cycle email digest: "You received 4 new feedback notes this week. Log in to review them." No note content included in email.

Should
Admin controls
Content moderation

Admin can flag or remove a note reported as inappropriate without revealing the original author to anyone. A "report note" action is available to members.

Must
Workspace settings

Configure grace period for note edits, feedback note character limit (default 400), available tags, and whether reactions are enabled.

Should

Board UI concept

Each team member gets a column. You can write on others' columns but not your own. Notes are sticky, draggable, and colour-coded by type.

Anonymity enforcement

Anonymity is enforced at multiple layers — not just visually hidden.

Server-side author storage only

The author's user ID is stored in the database but never included in the API response payload for the note endpoint. The recipient's client never receives this field.

Timing obfuscation

Note created_at timestamps are rounded to the nearest hour in the recipient's view, preventing identification by timing patterns.

Minimum note threshold for analytics

Tag/theme breakdowns in analytics are only shown if at least 3 notes exist, preventing a single author from being inferred through process of elimination.

Key user flows

Step-by-step paths for the two most important journeys in the product.

Flow A — Admin sets up a team
1

Admin signs in

Authenticates with email/password or magic link. Directed to the admin dashboard.

2

Creates a team

Clicks "+ New team", enters a team name (e.g., "Frontend team"), and saves. A new board is provisioned immediately.

3

Assigns members

Searches registered users by name/email and adds them. Optionally invites new users via email. Each added member gets a column on the board.

4

Opens a feedback cycle

Names the cycle (e.g., "Sprint 5"), sets optional start/end dates, and activates it. Members are notified in-app and by email digest.

5

Shares board link

Admin copies a shareable board link for the team to bookmark. Members log in and land directly on the team board.

Flow B — Member gives and actions feedback
1

Member opens the team board

Sees all team members as columns. Their own column is visually distinct (locked for writing). Others' columns have "+ Add feedback" buttons.

2

Writes a note on a colleague's column

Clicks "+ Add feedback" on Sam's column. A modal opens: selects note type (yellow/green/blue), adds optional tags, writes the note (max 400 chars), and submits. No name is requested or shown.

3

Recipient views their column

The note appears on Sam's column showing content, type colour, and tags — but zero authorship indication. Sam sees "Anonymous" where an author would normally appear.

4

Sam actions the note

Sam reads the feedback, takes action in their work, and clicks the checkmark to mark the note as "Done". The note collapses with a green completion chip. The board updates in real time for all members.

5

Metrics reflect the action

Sam's completion rate increases on their dashboard. The team admin sees participation metrics update. At cycle close, aggregate tag data becomes available.

Member analytics view (personal)

What each member sees on their own dashboard — growth-oriented, never punitive.

Notes received over time

Line chart: notes received per cycle, showing engagement trend. Helps members understand if they're becoming more visible and valuable to the team.

Strength radar chart

Radar/spider chart across all 5 tag categories, showing where the member consistently receives positive (green) vs growth-area (blue) notes.

Action streak

Gamified streak counter: consecutive cycles where the member marked all received notes as done. Encourages follow-through on feedback.

AI summary (v2)

Claude-generated 3-sentence plain-language summary of the member's feedback patterns at cycle close. Private to the member only.

Delivery roadmap

Four phases from MVP to full analytics. Each phase ships independently.

Phase Scope Target Success metric
Phase 1 · MVP Auth, team creation, member assignment, kanban board, anonymous sticky notes, self-column lock, mark-done, in-app notifications Month 1 First internal team using the board daily
Phase 2 · Feedback quality Note type colours, tag system, drag-and-drop reorder, edit/delete grace period, email invite, email digest, content reporting Month 2 >80% of notes have a tag; 0 unresolved content reports
Phase 3 · Analytics Personal dashboard, team admin view, feedback cycles, participation metrics, export to PDF/CSV, streak counter Month 3 Admin runs at least one cycle export per team per month
Phase 4 · Intelligence AI theme summariser, reaction system, anonymous 1-to-1 reply threads, Slack/Teams notification integration Month 4+ Members rate AI summary usefulness >4/5

Manager-visible individual ratings
Public leaderboards
Performance scoring / ratings

Cross-team comparisons
Manager-assigned mandatory feedback

Non-functional requirements

Quality constraints that cut across all features.

Security & privacy
  • Author ID never in API response
  • RLS: members only see own team data
  • HTTPS enforced everywhere
  • JWT expiry: 7 days / 30 days remember-me
  • Audit log for admin actions
Performance
  • Board load <1.5s for teams up to 20 members
  • Note submission <300ms perceived
  • Real-time board updates via WebSocket
  • Optimistic UI for drag-and-drop
Responsiveness
  • Board is horizontally scrollable on mobile
  • Note writing modal is mobile-optimised
  • Dashboard charts resize to container
  • Touch drag-and-drop supported
Accessibility
  • WCAG 2.1 AA for all interactive elements
  • Keyboard navigable board and notes
  • Screen reader announcements on note add
  • Colour not the only differentiator for note types
Stack alignment
  • Next.js frontend (Vercel deploy)
  • Supabase (PostgreSQL + RLS + Auth)
  • shadcn/ui components
  • dnd-kit for drag-and-drop
  • Supabase Realtime for live updates
Data & retention
  • Notes retained indefinitely by default
  • Admin can archive closed cycles
  • GDPR: member data deletion on request
  • No third-party analytics SDKs in v1

Open questions for next review

Should admins be able to see individual note authors?

There's a case for this in content moderation. However it risks a chilling effect on honest feedback. Recommend: admin can see author only after a formal report is filed, with a confirmation step.

Minimum team size for anonymity?

On a team of 2, any note is de-facto attributable. Should there be a minimum of 4 members before the feedback board activates?

Can members give feedback outside active cycles?

Always-on vs cycle-gated feedback both have merits. Always-on is more useful; cycles drive participation spikes. Propose always-on with optional cycles layered on top.
